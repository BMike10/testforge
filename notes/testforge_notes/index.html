    🛠️ TestForge: Scientifically Validated AI Test Generation¶

    TestForge is a high-integrity CLI framework that combines Deterministic Static Analysis with Agentic AI (powered by Aider) to generate, validate, and harden Python unit test suites.

    Moving beyond simple line coverage and standard AI code generation, TestForge employs topological mapping, complexity metrics, simulated symbolic execution, and mutation testing to ensure generated tests are structurally sound, behaviorally rigorous, and enterprise-ready.

    🔬 Scientific Core & Methodology¶

    TestForge operates on a continuous Research -> Plan -> Generate -> Validate lifecycle, strictly grounded in computer science principles:

    1. Topological Dependency Resolution (Bottom-Up)
      • Concept: Builds a Directed Acyclic Graph (DAG) of the codebase using Abstract Syntax Trees (AST).
      • Execution: Tests "leaf" modules (components with no dependencies) first. When testing high-level modules, the AI knows exactly what to mock based on the dependency graph, preventing "context bloat" and circular reasoning.
    2. McCabe Cyclomatic Complexity & Risk Prioritization
      • Concept: Deterministically calculates the complexity of every function/class using the graph-theoretic formula $M = E - N + 2P$.
      • Execution: The AI is explicitly instructed to prioritize high-complexity paths, ensuring the most fragile and dangerous code receives the most rigorous testing.
    3. Boundary Value Analysis (BVA) & DSE Simulation
      • Concept: Prevents the AI from "guessing" superficial test inputs.
      • Execution: TestForge forces the AI to define equivalence classes and explicitly test the $n-1$, $n$, and $n+1$ boundaries. Furthermore, it prompts the AI to simulate Dynamic Symbolic Execution (DSE) by mapping out logical branches and explicitly generating inputs to satisfy each unique path.
    4. Mutation-Guided Hardening
      • Concept: Line coverage is a "weak" metric. TestForge uses mutmut to inject semantic faults (mutants) into the source code.
      • Execution: If a test suite passes but a mutant survives, the test is empirically weak. TestForge identifies the surviving mutant and autonomously tasks the AI with hardening the assertions to "kill" it.

    ⚙️ Enterprise Configuration & Dynamic Routing¶

    TestForge is built to adapt to your company's specific engineering standards and infrastructure limits. It is highly configurable and model-agnostic.

    • Dynamic Model Routing: You are not locked into a single LLM for the entire pipeline. TestForge allows you to route different phases to different models. For example, you can use a fast, cost-effective model (or a local model) for Phase 1 (Architectural Scanning) and switch to a heavy reasoning model for Phase 3 (Test Generation).
    • Template-Driven Configuration: Every phase and LLM interaction is based on explicit prompt templates. Developers can easily override these templates to enforce internal coding guidelines, specific mocking libraries, or custom assertions.
    • Transparent Variable Injection: All variables available to the prompt templates (e.g., {{ file_path }}, {{ ast_complexity_score }}, {{ dependency_graph }}) are strictly documented in the Developer Guide.
    • Strict Privacy Mode (Local LLMs): Natively supports local inference via Ollama or vLLM. Enterprises can run architectural scanning entirely on local hardware to prevent proprietary IP from leaving the network.

    🛡️ System Safety & State Management¶

    • Agentic Context Isolation: Prevents LLM hallucination through strict memory management. The AI only "sees" the target file, the project's architecture map, and a deterministic summary of its immediate dependencies.
    • The Execution Ledger (State Checkpointing): TestForge maintains a strict pipeline plan saved in a cached state file. If the CLI is interrupted, hits an API error, or is paused by the user, it can resume exactly where it left off without duplicating work or API costs.
    • Financial Safety & Token Management: * Calculates the DAG, counts tokens, and provides an estimated API cost via a --dry-run command before execution.
      • Implements a strict max_retries circuit breaker. If a test fails after $X$ attempts, it is logged and bypassed, preventing infinite loop API billing.
    • Fixture Isolation (The conftest.py Shield): The AI is strictly prohibited from editing global conftest.py files during active unit testing to prevent a "global blast radius." Instead, it generates local fixtures. A separate testforge refactor-fixtures command safely elevates duplicated local fixtures to the global conftest.py.

    🚀 The 3-Phase Execution Pipeline¶

    Phase 1: Architectural Topography & Documentation¶

    Before writing tests, TestForge builds a complete mental model of the codebase.

    1. Project Architecture Generation:
      • Deterministic Scan: Uses Python AST tools to map classes, functions, and imports.
      • Simple Mode: Combines static analysis with a quick LLM scan to generate a high-level project map.
      • Complete Mode (Iterative): Iterating file-by-file (skipping utilities like __init__.py), it uses the AI to document specific modules deeply. This layered approach prevents overwhelming the LLM's context window.
    2. Codebase Documentation (Optional/Pre-requisite):
      • Analyzes the requested codebase to write or improve documentation (e.g., Google Docstrings). Generates an intermediate "Doc-Plan" for human review before applying changes.

    Phase 2: Strategic Test Planning¶

    With the map established, TestForge creates a deterministic battle plan. Note: All plans generated in this phase are output as markdown files. TestForge can be configured to block execution here, allowing the developer to review and manually edit the AI's strategy before generating code.

    1. Global Suite Strategy: Generates a comprehensive plan detailing how the codebase will be tested, identifying critical paths, maximum complexity areas, and shared fixture strategies.
    2. Unit Strategy (Micro-Plans): Generates a specific, markdown-based strategy file for an individual module or file. This acts as the strict blueprint the AI must follow when generating code.

    Phase 3: Agentic Execution & Validation Loop¶

    The active generation phase where tests are written, run, and iteratively fixed based on the Phase 2 strategy.

    • Execution Modes:
      • Single-File Mode: Executes the unit strategy for one specific file using HitL (Human-in-the-Loop) pauses for review.
      • Suite Iteration Mode: Uses the Global Suite Strategy to systematically iterate over the entire codebase.
      • CI/CD Headless Mode: Built for GitHub Actions/Jenkins. When triggered by a PR, TestForge analyzes the diff, generates tests only for changed files and their DAG dependents, and pushes a commit directly without human intervention.
    • The Validation Loop (Self-Healing):
      1. AI generates test -> pytest executed -> Coverage/Mutation checked.
      2. If the test fails, errors out, or hallucinates, the error traceback is fed back to the AI alongside the original strategy.
      3. The AI corrects the test. This loops until the test passes perfectly or hits the circuit breaker.
